# 연동 전략: 커널-모듈 비대칭 구조

> **궁극 목표**: 로그를 모아서 클론을 만든다.
> **핵심**: 피지컬 AGI로 가는 가장 현실적인 경로는 직접 만드는 것이 아니라, 잘 구축된 인프라를 "제어 레이어"로 연결하는 것이다.
> **작성일**: 2026-02-12 | **갱신**: 2026-02-13

---

## 1. 왜 잘 구축된 앱과 연동하는 것이 전략적으로 유리한가?

### 결제선생 사례

이 회사는:
- 수납 인프라 완성
- PG 안정화
- 정산 구조 확립
- 법적/회계 리스크 처리 완료

이걸 직접 만들면:
- 1~2년 소요
- 법/정산 리스크
- 고객 신뢰 구축 비용

→ **무소유 전략과 충돌**

## 2. 피지컬 AGI 관점에서의 의미

피지컬 AGI는 3가지가 필요하다:

1. **상태 데이터**
2. **자원 데이터**
3. **실행 능력**

결제선생은 이미 자원 흐름(돈)이라는 현실 데이터를 제공한다.

> 결제 연동 = 물리 세계 자원 제어의 시작

### 궁극 목표: 클론

```
학원들 → 수업 → 로그 생산 → 학생에게 누적 → 임계점 → 클론 탄생 → 피지컬 AI
```

피지컬 AI는 **초개인화 로그**를 가져야 한다. 초개인화 = 한 사람의 로그가 충분히 쌓여야 가능. 온리쌤은 학원 관리 도구가 아니라, **클론을 만들기 위한 로그 수집기**다.

### 클론에 필요한 5대 로그

| # | 로그 유형 | 소스 | 피지컬 AI에서의 의미 |
|---|----------|------|-------------------|
| 1 | **움직임** | 유튜브 (비공개 링크) | 신체가 한 것 |
| 2 | **관찰** | 노션 (텍스트 피드백) | 코치가 본 것 |
| 3 | **빈도** | 출결선생 | 얼마나 자주 훈련하는가 |
| 4 | **지속** | 결제선생 | 얼마나 오래 지속하는가 |
| 5 | **패턴** | 상담선생 | 언제, 어떤 조합으로 |

> 5가지가 한 사람에게 충분히 쌓이면, 그 사람의 피지컬 클론이 가능해진다.

### 학생 중심 아키텍처

기존: **학원 → 학생** (학원이 학생을 관리)
전환: **학생 ← 여러 학원** (학생에게 로그가 수렴)

```
학원A (농구) ──→ 노션 + 유튜브 ──┐
학원B (수영) ──→ 노션 + 유튜브 ──┤──→ 학생 포트폴리오 ──→ 클론
학원C (수학) ──→ 노션 + 유튜브 ──┘
                                  ↑ 시간이 갈수록 누적
```

- 학원은 **모듈** — Truth를 생산하고 보낸다
- 학생은 **커널** — 모든 로그가 학생에게 수렴한다
- 학원을 옮기든, 졸업하든, **로그는 학생 것으로 남는다**
- `student_id`는 `org_id`를 초월하는 독립 엔티티여야 한다

## 3. 연동 전략의 핵심 원칙

### ① 우리는 커널, 그들은 모듈

| 선생 | 역할 | Truth | 데이터 |
|------|------|-------|--------|
| 결제선생 | 수납 | Payment Truth | 돈의 흐름 |
| 출결선생 | 출석 | Presence Truth | 존재의 상태 |
| 상담선생 | 스케줄 | Schedule Truth | 시간의 배치 |
| **AUTUS Kernel** | **정책 + 실행** | **Decision Truth** | **로그 기록** |

> **Truth는 그들, 의사결정은 우리.**

### ② 직접 구현하지 말 것

- 결제 기능 직접 구현 ❌
- PG 직접 연동 ❌
- 회계 처리 ❌

우리는:
- 결제 **상태를 읽고**
- **정책을 실행하고**
- **행동을 재배치한다.**

### ③ IOO 표준으로 감싼다

```
결제선생 API → Input
우리 정책   → Operation
카톡 발송/일정 수정 → Output
```

**모든 연동은 IOO로 감싸야 한다.**

## 4. 피지컬 AGI로 가는 단계별 연동 전략

### 단계 1 — 한 학원의 Truth 확보 ← 현재

결제/출석/스케줄 연동:
- 결석 알림, 미납 대응, 일정 변경
- 노션 + 유튜브 로그 적재 시작
- **학생 독립 엔티티 설계** (org를 초월)

> 핵심: 1단계에서 이미 로그 구조가 학생 중심이어야 한다

### 단계 2 — 여러 학원의 Truth 수렴

- 복수 학원 → 한 학생 포트폴리오
- 종목별 로그 크로스 분석
- 캘린더, 출입 시스템, QR/NFC, 메시징 확대

> 핵심: 학원이 많아질수록 한 학생의 로그가 두꺼워진다

### 단계 3 — IoT + 신체 센서 데이터

- 조명, 출입, 온도, 위치 감지
- 웨어러블 디바이스 연동
- 움직임 로그의 정밀도 상승

> 핵심: 유튜브 영상 + 센서 = 움직임 로그 완성

### 단계 4 — 클론 + 산업 확장

- 충분한 로그 → 피지컬 클론 생성
- 케어 산업 → 약물 기록, 병원 → EMR, 물류 → WMS
- **각 산업에서 같은 패턴**: 로그 수집 → 클론 → 피지컬 AI

> 이 단계에서 AGI는 "중앙 제어 레이어"가 된다.

## 7. 절대 원칙

> 직접 다 만들려는 순간 패배 확률이 급상승한다.
> 잘 만든 인프라를 먹고, 그 위에서 정책과 실행을 장악하는 것이 가장 현실적인 경로다.

### "우리가 절대 직접 만들지 않을 영역"

- PG/결제 인프라
- 정산/회계 처리
- 카카오 메시징 인프라
- 캘린더 인프라
- 출입/IoT 하드웨어

### "우리만 할 수 있는 영역"

- Encounter Kernel (존재의 상태 판단)
- Policy Engine (위험 감지 → 행동 결정)
- V-Index (물리 기반 가치 평가)
- IOO Trace (감사추적)
- Worker Gateway (실행 오케스트레이션)
- **Student Portfolio** (학생 중심 로그 누적)
- **Clone Engine** (로그 → 클론 변환)

## 8. 온리쌤의 실체

| 표면 (사용자가 보는 것) | 실체 (시스템이 하는 것) |
|----------------------|---------------------|
| 학원 관리 도구 | 피지컬 AI 데이터 파이프라인 |
| 출석 체크 | 존재 로그 (빈도) 수집 |
| 수납 관리 | 지속성 신호 (지속) 수집 |
| 수업 영상 (유튜브) | 움직임 로그 수집 |
| 코치 피드백 (노션) | 관찰 로그 수집 |
| 수업 스케줄 | 패턴 로그 수집 |
| V-Index 점수 | 클론의 첫 번째 파라미터 |

> 온리쌤은 학원 관리 SaaS가 아니라 **학생 중심 교육 이력 플랫폼**이다.
> 궁극적으로, **피지컬 AI의 데이터 레이어**다.
