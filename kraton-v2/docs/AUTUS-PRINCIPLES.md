# AUTUS 핵심 원리

## 왜 Amazon, Tesla, Palantir인가?

세 회사의 공통점: **인간의 판단을 시스템으로 대체**

- Amazon: 고객이 원하는 것을 인간이 추측 → **데이터가 말해줌**
- Tesla: 운전자가 판단 → **AI가 관찰하다가 대신함**
- Palantir: 분석가가 결정 → **시스템이 영향을 보여주고 결정을 기록함**

---

## 1. Amazon: Customer Obsession

### 핵심 통찰

> "고객 로그가 프로세스를 만든다. 사람이 프로세스를 설계하는 게 아니다."

### 원리

1. **모든 것은 고객 행동에서 시작**
   - 고객이 클릭 → 추천 알고리즘 작동
   - 고객이 불만 → 보상 프로세스 작동
   - 고객이 이탈 → 윈백 프로세스 작동

2. **Working Backwards**
   - "어떤 프로세스를 만들까?" (X)
   - "고객에게 어떤 결과가 필요한가?" → 역산해서 프로세스 도출 (O)

3. **Kill Culture**
   - 효과 없는 프로세스는 빨리 죽인다
   - 감정 없이, 데이터로 판단

### AUTUS 적용

```
고객 이벤트 발생
    ↓
OutcomeFact 생성 (결과 사실)
    ↓
매칭되는 정책 자동 실행
    ↓
효과 측정
    ↓
효과 없으면 Kill
```

**핵심 질문**: "이 프로세스는 어떤 고객 이벤트에서 시작되는가?"

## 2. Tesla FSD: Shadow Learning

### 핵심 통찰

> "바로 실행하지 않는다. 충분히 관찰하고, 신뢰가 쌓이면 자동화한다."

### 원리

1. **Shadow Mode**
   - 새 정책은 바로 실행하지 않음
   - "만약 이 정책이 실행됐다면?" 예측만 함
   - 실제 결과와 비교

2. **Confidence Accumulation**
   - 예측이 맞을 때마다 신뢰도 상승
   - 틀리면 하락
   - 충분한 관찰 (50회 이상)

3. **Promotion Threshold**
   - 신뢰도 70% → Candidate (주시)
   - 신뢰도 90% + 50회 관찰 → Promoted (자동 실행)
   - 언제든 Kill 가능

### AUTUS 적용

```
새 정책 등록
    ↓
Shadow Mode (실행 안 함, 관찰만)
    ↓
예측 vs 실제 비교
    ↓
신뢰도 축적
    ↓
임계값 도달 → 자동 Promotion
```

**핵심 질문**: "이 정책은 얼마나 관찰되었고, 신뢰도는 몇 %인가?"

## 3. Palantir: Decision OS

### 핵심 통찰

> "결정하기 전에 영향 범위를 본다. 결정은 불변 로그에 기록된다."

### 원리

1. **Ontology (온톨로지)**
   - 세상의 객체를 정의: Customer, Producer, Contract, TimeSlot
   - 객체 간 관계 정의: enrolled_in, taught_by, bound_to

2. **State Machine**
   - 계약은 상태를 가짐: S0(대기) → S1(접수) → ... → S9(종료)
   - 허용된 전이만 가능
   - 모든 전이는 기록됨

3. **Blast Radius**
   - 상태 전이 전에 영향 범위 계산
   - "이 결정이 몇 개의 계약에 영향을 주는가?"
   - "매출 영향은 얼마인가?"
   - 보고 나서 결정

4. **Immutable Log**
   - 모든 결정은 기록됨
   - 수정/삭제 불가 (Append-Only)
   - 누가, 언제, 왜, 무엇을 했는지

### AUTUS 적용

```
상태 전이 요청
    ↓
Blast Radius 계산
    ↓
영향 범위 표시 (계약 수, 고객 수, 매출)
    ↓
Owner 승인
    ↓
Immutable Log에 기록
    ↓
상태 전이 실행
```

**핵심 질문**: "이 결정의 영향 범위는 무엇이고, 누가 승인했는가?"

## 통합: AUTUS 원리

### 단일 흐름

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  고객 이벤트 발생 (Amazon)                                       │
│       ↓                                                          │
│  OutcomeFact 생성                                                │
│       ↓                                                          │
│  정책 매칭                                                        │
│       ↓                                                          │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ Shadow Mode (Tesla)                                          ││
│  │   - 실행 안 함                                                ││
│  │   - 예측만 기록                                               ││
│  │   - 신뢰도 축적                                               ││
│  │   - 임계값 도달 시 Promotion                                  ││
│  └─────────────────────────────────────────────────────────────┘│
│       ↓                                                          │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ Promoted 정책이면 실행                                        ││
│  │   - Blast Radius 계산 (Palantir)                             ││
│  │   - 영향 범위 표시                                            ││
│  │   - 승인 필요하면 Owner에게                                   ││
│  │   - Immutable Log 기록                                        ││
│  └─────────────────────────────────────────────────────────────┘│
│       ↓                                                          │
│  상태 전이 (S0 → S1 → ... → S9)                                  │
│       ↓                                                          │
│  효과 측정                                                        │
│       ↓                                                          │
│  효과 없으면 Kill (Amazon)                                       │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 5가지 핵심 규칙

1. **No Free Text**: 입력은 버튼/선택지만. 자유 입력 없음.
2. **Outcome First**: 프로세스가 아니라 결과(OutcomeFact)가 먼저.
3. **Shadow Before Execute**: 새 정책은 관찰부터.
4. **Blast Radius Before Decide**: 결정 전에 영향 범위 확인.
5. **Immutable Log**: 모든 결정은 불변 기록.

## 역할별 원리 적용

### Owner (원장)

- **본다**: Blast Radius, Kill 대상 정책
- **결정한다**: 승인/거절/보류
- **책임진다**: 모든 결정이 로그에 기록됨

### Manager (관리자)

- **제안한다**: 새 정책을 Shadow로 등록
- **모니터한다**: 정책 신뢰도 추적
- **보고한다**: Owner에게 Promotion 추천

### Staff (코치/직원)

- **기록한다**: 고객 이벤트 발생 (출석, 피드백 등)
- **실행한다**: Promoted 정책 수행

### Customer (고객)

- **발생시킨다**: 이벤트 (모든 프로세스의 시작점)
- **응답한다**: 선택지로만 (No Free Text)
